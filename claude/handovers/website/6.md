## Schema and architecture notes

- **STATE_KEY** is still `'cnc-shopping-state-v5'`. Was tempted to bump
  during the override removal but the loader's `Object.assign(fallback, parsed)`
  pattern makes additive changes safe. The new fields are:
  - `userPriceObservations: { "<itemCode>::<shopId>": { price, currency,
    eta_days, in_stock, ts } }`
  - `userShops: { "<itemCode>": [ { id: "__addedshop_<item>_<hex>",
    shop_label, url, price, currency, eta_days, shipping_cost, in_stock,
    ts } ] }` — synth shop ids stay in localStorage; merge promotes them
    to stable `user-<slug>` ids in shops.json.
  - `userEans: { "<itemCode>::<shopId>": "<ean>" }`
  - `openAdd: { "<parentCode>": 'alt' | 'shop' }` — replaces the old
    `expandedItems` boolean (with one-shot migration).
  - `userAlternatives` unchanged.
  - `state.itemOverride` — **deprecated**; one-shot migration on load
    moves any leftover `state.itemOverride[code].price` into
    `userPriceObservations[code::firstScopedShop]` then clears it.
- **Constants**: `ADDED_SHOP_PREFIX = '__addedshop_'` (user-added
  shops on an existing item). `USER_SHOP_PREFIX = '__user_'` (the
  synth shop attached to a user-added alternative). Distinct prefixes
  matter — each has its own sync function and re-runs clear-by-prefix.
- **Sync order** in `init()` is significant:
  ```
  syncUserAlternativesIntoData()
  syncUserShopsIntoData()
  migrateItemOverridesToUserPrices()
  migrateExpandedItemsToOpenAdd()
  syncUserPriceObservationsIntoData()
  applyUserEansToData()
  ```
  In particular `applyUserEansToData()` resets each `entry.ean` to its
  seed via `entry._seedEan` (snapshotted at ingest) and then overlays
  state.userEans — so `Clear` returns to the seed value cleanly.
- **`prices.json` `$schema_version`** still 2; observations carry
  `technique: 'jsonld' | 'opengraph' | 'affiliate-api' | 'manual'`.
  Manual user observations get `technique: 'manual'`.
- **Head row order** (final): checkbox → thumb → `[code-button · alts-select · qty]`
  → shop-select + meta(stock·eta·ship) → `↗ Visit` → EAN button →
  history sparkline → `+ Alt` → `+ Shop` → price (right-aligned).
- **History popover** was preserved on the slim head row via
  `renderHeadHistoryControl(effective)` — opens for whichever shop is
  currently chosen in the dropdown. Validate buttons inside the popover
  still work (SL-8.i).

## The repo-is-truth workflow (SL-10.T/U/V)

End-to-end this is **one command**:

```
python3 tools/dev.py
```

That spawns:

- `mkdocs serve -a 127.0.0.1:8000`
- `python3 tools/refresh_server.py --port 8765`

The shopping page polls `GET /api/health`, sees `write_through: true`,
and from then on every save action POSTs the relevant state slice to
`POST /api/save-manual`. The handler invokes
`merge_user_data.merge_state(state, dry_run=False)` which writes to
`docs/data/shops.json`, `items.json`, `prices.json` directly and replies
with `{summary, consumed, id_translations}`. The page clears the consumed
keys from `state.user*` buckets so the next render reads from JSON only
(no double-overlay), rewrites `state.shopOverride` references via
`id_translations` (so the dropdown still selects the user's shop after
its synth id is promoted to a stable repo id), and pops a "Saved to
repo" badge.

If the helper isn't running:
1. Page silently falls back to localStorage.
2. User clicks **"↓ Export data"** in the page header → downloads a
   timestamped JSON.
3. `python3 tools/merge_user_data.py path/to/exported.json` folds it
   into `docs/data/*.json`.

The CLI is now a thin wrapper around the library entry point
`merge_state(state, *, data_dir=DATA_DIR, dry_run=False)`. Both the
write-through endpoint and the CLI go through it, so behavior stays
identical.
