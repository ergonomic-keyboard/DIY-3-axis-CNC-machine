## Schema changes the next chat must know about

### `docs/data/items.json` — `$schema_version: 2`

```jsonc
{
  "$schema_version": 2,
  "configurations": [
    // Each entry is a built-in named bundle:
    //   { "id": "<slug>", "label": "<human>", "uses": { "<parent-code>": "<alt-code>" } }
    // The Default (stock build) bundle is implicit — never list it here.
  ],
  "categories": [ /* … */ ],
  "items": [
    {
      "code": "E20",
      "name": "Makita RT0700C (milling router)",
      // …
      "alternatives": [
        // Each alternative has its own unique `code`. That code IS used as
        // item_code in prices.json — bots fetch alternatives the same way.
        // `qty`, `image` fall back to the parent when omitted.
        { "code": "E20-katsu", "name": "Katsu 1/4\" trim router (budget alternative)" }
      ]
    }
  ]
}
```

One demonstration alternative (`E20-katsu`) is in the file so the UI is
testable. No price entries for it in `prices.json` — that's intentional;
users add a shop via the existing override form, or a bot will fetch
it when SL-9.d/the bot bundle land.

### `localStorage['cnc-shopping-state-v3']` (key bumped from v2)

```jsonc
{
  "version": 3,
  "country": "NL",
  "selected":     { /* parentCode → bool */ },
  "shopOverride": { /* effectiveCode → shopId — keyed by the visible product, not the parent */ },
  "itemOverride": { /* effectiveCode → custom-shop object — same keying as shopOverride */ },
  "defaultsApplied": true,
  "thumbRem": 3,
  "activeConfig": "default" | "<config-id>" | "custom",
  "customUses":   { /* parentCode → alternativeCode */ },
  "savedConfigurations": [ /* same shape as items.json configurations[] */ ]
}
```

Bump again on any further schema-affecting change (notably when
SL-8.d's follow-up adds `state.userAlternatives`, or SL-8.f stores
per-observation `validated_at` in app state rather than data).

## Architecture notes from the in-flight refactor

- **`STATE_KEY`** is `'cnc-shopping-state-v3'`. Bump on any
  schema-affecting state change.
- The shopping page is rendered by a single `render()` call that
  rebuilds the whole tree. There's no diffing; the slider live-update
  uses direct DOM mutation on `--shopping-thumb-size` to avoid a full
  re-render during drag, but every other state change re-renders the
  page.
- **Effective code resolution.** `effectiveCodeFor(parentCode)` is the
  single source of truth for "which product fills this slot right now."
  All pricing helpers (`buildOrder`, `buildSectionSummary`) iterate
  `state.selected` by parent code and resolve to effective code before
  looking up shops/prices/qty. Per-product state (shop pick, custom
  shop override) is keyed by effective code; per-slot state (selection)
  is keyed by parent code.
- **Inline form is shared.** `renderInlineProductForm` is the only
  builder; `renderItemOverride` is a thin caller. SL-8.d follow-up's
  "Add alternative product" should be a second caller, with a different
  `onSave` that writes to `state.userAlternatives` instead of
  `state.itemOverride`.
- **Side pane is gone.** `.shopping-app` is `display: flex; flex-direction: column`. The summary lives at the bottom, full-width, with a CSS-grid of shop groups inside it.
