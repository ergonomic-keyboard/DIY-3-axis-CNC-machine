## What changed in this session

### `align_outline.py` — vertex mode added

The interactive polygon editor was edge-only. Now **P** toggles a
vertex mode:

- **Click** selects the nearest polygon vertex (highlighted with a red
  ring + `V{i}` label).
- **← → ↑ ↓** (Shift = fine 0.1 mm) moves the selected vertex in true
  X / Z (not edge-perpendicular like edge mode).
- **M** then click: snaps the selected vertex to the nearest "STL
  point" — either an endpoint of a plastic-boundary segment or a hole
  centre (both precomputed once at startup, drawn as cyan dots while in
  the mode).
- Moving or snapping a vertex auto-clears the H/V constraint on its
  two adjacent edges, otherwise `apply_all_constraints` partly undoes
  the move on save.
- Edge-only keys (H / V / N / G / 1 / 2 / F) become no-ops in vertex
  mode with a one-line hint. Banner and status line update per mode.
- Esc cancels the vertex-snap manual-pick.

The docstring at the top of the file lists all keys.

### `build_model.py` — M5 nut overlay on every render

Two helpers added near the renderers:

- `_detect_slot_nut_centres(outline_xz_mm)` — finds keyhole slots by
  their short horizontal "ledge" edges (length 1 – 3 mm). For each Z
  group of ledges it sorts them by X and pairs consecutive ones; the
  slot centre is the midpoint between the two pair midpoints. Slot
  orientation (open up vs. down) is decided from whether the ledge Z is
  above or below the plate's mid-Z; the nut centre is then offset by
  AC/2 into the wide part.
- `_draw_m5_nuts(ax, centres)` — draws an M5 DIN 934 hex outline
  (AF = 8 mm, AC = 9.24 mm) in **captive orientation** (flats vertical
  / parallel to the slot walls, AC vertical). Red lines + centre `+`.

Both `render_plan()` and `render_overlay()` call them, so the nuts
appear in `*_metal_plan.png` and `*_metal_vs_plastic.png`. No new CLI
flag needed — it just runs.

### Gotcha worth flagging: keyhole slot traversal direction

build123d's `Wire.combine` will silently produce an **open** wire (and
`make_face()` then errors `Face can only be created with closed
wires`) if a keyhole slot is traversed in the wrong direction along
the polygon — specifically when the bottom-edge segments on either
side of the slot opening sweep overlapping X ranges.

The original trace had this for the bottom-right slot (BR): edges
e34 and e42 both lay on Z = −5.49 and overlapped 13 mm in X. It
"happened to work" because `Wire.combine` extracted a closed sub-wire
with one stray edge. Once the slot was resized smaller, the overlap
shrank to 6 mm and the algorithm produced one big open wire instead —
breaking the build.

Fix is structural, not numeric: **reverse the slot's interior vertex
order** so the slot is entered from the wall closer (along the
polygon traversal) to the previous boundary vertex. For BR this meant
swapping the vertex pairs (v35, v42), (v36, v41), (v37, v40),
(v38, v39). Physical shape unchanged, traversal opposite.

When tracing future parts with keyhole slots, double-check that each
slot is entered from the "near" wall, not across the slot opening.
The TL, TR and BL slots in this part were already correct.
