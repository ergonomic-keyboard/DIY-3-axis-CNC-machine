## Key change since 1.md: rectify.py is now axis-agnostic

`Z_MOTOR_MOUNT.stl` was a red herring — the user clarified the correct
STL is **P07 / `docs/stl_files/router/CARRIAGE.stl`**
(BOM: P07 = Carriage). Its bbox is **40 × 153 × 239 mm**; the face is
perpendicular to **X** (40 mm thin), with **65 X-axis face holes**.

Previously `rectify.py` assumed the face is perpendicular to Y. To handle
this part and any future parts where the face is X- or Z-perpendicular,
the pipeline now does an **STL pre-rotation**:

| File | What changed |
| --- | --- |
| `rectify.py` | new `--face-axis {auto,X,Y,Z}` flag. `auto` picks the axis with the most small-radius (≤ 6 mm) holes. If face_axis ≠ Y, the script writes a rotated copy to `<example>/2_flattened_image/stl_face_y.stl` and runs `extract_holes` on that. Everything downstream sees "Y is the face axis" as before. `holes_from_stl.json` now records `face_axis_original`, `effective_stl`, `source_stl`. The `_rect.json` `"stl"` field now points at `stl_face_y.stl` (used by `build_model.py` + `align_outline.py` for the overlay STL projection — they keep working unchanged). |
| `rectify.py` | the canvas size is now **derived from the plate's STL bbox**, not the warped photo's bbox. Previously when calibration clicks were imprecise the canvas could collapse to a ~50 × 160 mm strip ; now it's always plate-sized. `--max-px` default also raised from 40 Mpx → 200 Mpx. |
| `rectify.py` | `plt.close("all")` before the photo-click window — fixes a Qt/Wayland issue where the hole-picker window stayed on top, swallowing clicks intended for the photo. |
| `rectify.py` | calibration picks are persisted to `_rect.json` **before** the canvas / homography check — if the run aborts at that step, the next run re-loads them. |
| `rectify.py` | if the user re-orders the same 4 holes in the picker, cached photo clicks are **re-permuted** to match the new order instead of being discarded. |

Rotation helper details (`_permute_triangles` + `write_rotated_stl`):

- `face_axis="Z"`: swap Y↔Z (screenshot horiz = old X stays, screenshot
  vert = old Y becomes new Z, face = new Y = old Z).
- `face_axis="X"`: swap X↔Y (screenshot horiz = old Y becomes new X,
  screenshot vert = old Z stays, face = new Y = old X).
- `face_axis="Y"`: identity.

After rotation, `numpy-stl` `update_normals()` recomputes face normals so
OCCT-based loaders don't see inverted faces.

Caveat: this metal plate is **p1of2** — the metal version splits the
carriage into 2 pieces. **Some auto-picked holes may not exist on the
metal plate**. The user must toggle to 4 holes that are visible on the
physical plate in the screenshot. Caching now re-permutes cached clicks
so order changes don't lose work.
