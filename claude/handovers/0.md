# Handover 0 — Shopping-links feature

## What the next chat is being asked to do

Implement the requirements in `claude/objectives/shopping_links.md` (SL-0.a → SL-2.d). Read that file first — it is the source of truth and supersedes anything below.

In short: surface per-component buy links on the site, with per-shop price / availability / shipping ETA, support multiple countries (start NL-only), let the user pick which components to buy and which shop to use, and compute a combined total including shipping with free-shipping thresholds honoured.

## Repo orientation (1-minute version)

- **Static site only.** Built with MkDocs + Material theme. No backend, no DB. Source pages live in `docs/`; nav lives in `mkdocs.yml`. Deploy is via GitHub Actions on push to `main` (`.github/workflows/`).
- **Bill of materials** is `docs/BILLOFMATERIAL.md` — a hand-written Markdown table keyed by item codes (`P01`, `S01`, `N01`, `E01`, `T01`, `O01`, …). This is what shopping links must attach to. Item codes are the natural join key.
- **Custom assets** already wired: `docs/assets/stylesheets/` and `docs/assets/javascripts/`, registered in `mkdocs.yml` under `extra_css` / `extra_javascript`. Use the same pattern for any new client-side code.
- **Local dev**: see `developers.md`. `pip install mkdocs-material`, then `NO_MKDOCS_2_WARNING=1 mkdocs serve`.

## Settle this before writing code

The objective is silent on *how* the prices appear. The answer reshapes the whole implementation. Decide with the user up front — don't guess:

1. **Data source.** Three plausible shapes:
   - **Static / baked**: a YAML or JSON file in the repo lists shops, prices, shipping, ETA per item. Updated by hand or by a maintenance script. Pro: zero runtime infra, deterministic, works on GitHub Pages. Con: prices go stale.
   - **Client-side fetched**: JS hits shop APIs / scrapes at page load. Pro: live. Con: CORS, rate-limits, API keys can't be secret on a static site, brittle, slow.
   - **Hybrid**: static snapshot in repo + a scheduled GitHub Action that refreshes it. Pro: live-ish, no client secrets. Con: more moving parts.
   - Default recommendation: **static snapshot**, with a clear schema and an `last_updated_at` field surfaced in the UI. Upgrade to hybrid later if needed.

2. **Where the UI lives.** Options: a new dedicated "Shopping" page; inline columns added to the BOM table; or a modal/drawer on the BOM page. Pick one before designing the schema.

3. **State persistence.** Component selection + shop overrides per-user need to live somewhere. `localStorage` is the obvious answer for a static site — same pattern as the TOC toggle (`docs/assets/javascripts/toc-toggle.js`).

## Constraints worth pinning

- **NL first** (SL-1.a). Don't generalise the data model away from NL until at least one country works end-to-end. But the data shape *should* anticipate multiple countries — country is a key, not a hardcoded column.
- **"All countries" option** (SL-1.c) is a price-comparison view, not its own country — it needs to fan out across the data.
- **Free-shipping thresholds** (SL-2.b) belong on the *shop*, not the item — model accordingly.
- **User-supplied overrides** (SL-2.c, SL-2.d) need to round-trip without losing data on a page reload. Include an EAN or product link as the override identity.
